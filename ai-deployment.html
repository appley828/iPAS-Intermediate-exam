🔧 模型壓縮三大技術（核心考點）
讓 AI 模型在有限資源環境下運行：剪枝 → 量化 → 知識蒸餾
剪枝（Pruning）
移除重要性低的連結或神經元，減少模型大小。
• 非結構化：移除個別權重（稀疏矩陣）
• 結構化：移除整個濾波器（易於加速）
通常壓縮 50~90%，精度損失 <2%
▶ 考點：移除低重要性連結/神經元
量化（Quantization）— 必考數字
FP32 → INT8：記憶體節省 75%（4 bytes→1 byte）
推論速度快 2~4×
• 後訓練量化（PTQ）：訓練後直接量化，快
• 量化感知訓練（QAT）：精度更高
▶ 考點：INT8 比 FP32 省 75% 記憶體
知識蒸餾（Knowledge Distillation）
大模型 Teacher → 小模型 Student
Student 學習 Teacher 的軟標籤（Softmax 輸出），而非硬標籤。溫度 T>1 讓輸出更平滑，暴露更多「暗知識」。
▶ 考點：Teacher 大模型 → Student 小模型
📌 iPAS 必記：FP32=32bit=4bytes → INT8=8bit=1byte，節省 75%（=(4-1)/4）。推論速度通常提升 2~4 倍。這是最高頻的量化考點！
📡 邊緣運算 vs 雲端部署
選擇部署環境：延遲、頻寬、隱私與成本的權衡
比較維度邊緣運算（Edge Computing）雲端運算（Cloud Computing）
推論延遲✅ 毫秒級，無網路往返⚠️ 50~200ms+（需網路傳輸）
網路依賴✅ 離線可運作（斷網仍推論）❌ 需穩定網路
資料隱私✅ 資料不離裝置⚠️ 資料上傳雲端（需加密）
算力❌ 受限於裝置硬體（需壓縮模型）✅ GPU/TPU叢集可無限擴展
模型更新⚠️ 需 OTA（Over-The-Air）更新✅ 即時更新，無需更新裝置
典型情境自動駕駛、醫療裝置、IoT、工廠大規模訓練、跨裝置服務、推薦
情境題關鍵字
「即時偵測行人」「隧道無網路」「保護病患隱私」→ 邊緣運算
「根據全站資料調整推薦」「大規模模型訓練」→ 雲端
三大邊緣優勢：低延遲、離線、隱私
▶ 低延遲/離線/隱私 → 邊緣
霧運算（Fog Computing）
邊緣與雲端的中間層，在本地網路（工廠閘道器）先做初步處理，過濾後才上傳雲端。減少雲端頻寬，比純邊緣算力更強。介於 Edge 和 Cloud 之間。
▶ 考點：霧運算 = 中間層
邊緣 AI 必備：模型壓縮
邊緣裝置（Raspberry Pi、手機、Jetson）算力有限，必搭配：量化 INT8（省記憶體）、剪枝（省計算）、知識蒸餾（輕量模型）。目標：5W 功耗內完成推論。
▶ 邊緣 AI = 壓縮 + 低功耗
✂️ 3D 神經網路剪枝視覺化
稠密網路 → 剪枝稀疏網路動畫（可拖曳旋轉）
✂️ 模型剪枝 3D · 拖曳旋轉

⚙️ MLOps 流水線（CI / CD / CT）
機器學習的 DevOps：持續集成 → 持續部署 → 持續訓練

持續集成

→

持續部署
Docker 打包、藍綠部署、Canary Release（5%流量→全量）
→

持續訓練

→
監控
持續監控
效能指標監控、Data/Concept Drift 偵測、觸發告警
CI（持續集成）
程式碼提交後自動執行：程式碼風格（Linting）、資料 Schema 驗證、模型單元測試。工具：GitHub Actions、Jenkins、GitLab CI。
▶ 考點：CI = 自動測試品質
CD 部署策略
藍綠部署：兩套環境，一次切換 100% 流量，零停機。Canary Release：先 5~10% 流量走新版，確認正常後全量切換，降低風險。
▶ 考點：Canary = 少量流量先測試
CT（持續訓練）— MLOps 特有
傳統 DevOps 只有 CI+CD，MLOps 加了第三個 C：CT。當偵測到 Data Drift 時，自動觸發再訓練流水線。ML 模型會因數據分佈改變而退化，程式碼不會。
▶ 考點：CT = MLOps 獨有
💡 考試情境：「輸入資料分佈與訓練資料不同，應觸發哪個 MLOps 流程？」→ CT（持續訓練）。「零停機部署新版模型」→ 藍綠部署。「先讓少量用戶使用新模型」→ Canary Release。

📊 模型監控（Data Drift / Concept Drift）
AI 系統上線後的效能退化偵測
資料漂移（Data Drift）
輸入特徵 X 的統計分佈改變，但任務定義不變。
例：2020年訓練的詐欺偵測模型，2024年詐欺手法演變 → 輸入模式不同。
偵測：KS-test、PSI（Population Stability Index）。
▶ 輸入特徵分佈改變 → 觸發 CT
概念漂移（Concept Drift）
X→Y 的映射關係改變，即任務定義本身改變。
例：「垃圾郵件」定義隨新型詐騙演進。即使輸入分佈沒變，模型也會失準。
比 Data Drift 更難偵測，需標記新資料確認。
▶ X→Y 映射改變（任務定義變）
模型退化監控策略
設定效能警戒線（如 F1 < 0.85 觸發告警）→ 自動觸發 CT 再訓練 → 評估新模型是否優於現役 → 通過則部署（CD）。
工具：MLflow、Evidently AI、WhyLogs。
▶ 效能下滑 → 觸發 CT
漂移類型什麼改變偵測方法應對
Data Drift輸入特徵 X 的分佈KS-test、PSI用新資料再訓練
Concept DriftX→Y 映射關係需新標記資料確認重新標記 + 再訓練
Label Drift輸出標籤 Y 的分佈預測分佈監控確認業務定義改變
Prediction Drift模型輸出分佈預測值統計分析檢查是否為 Data Drift
🔒 差分隱私（Differential Privacy）
在不洩露個人記錄的前提下學習統計規律
// ε-差分隱私定義
P[M(D) ∈ S] ≤ eε × P[M(D') ∈ S]

// ε（隱私預算）越小 = 保護越強（噪聲越大）
ε → 0: 強保護（高噪聲，效用低）
ε → ∞: 弱保護（低噪聲，效用高）

// 兩大噪聲機制
拉普拉斯機制: Noise ~ Lap(Δf / ε) // 數值查詢
高斯機制: Noise ~ N(0, σ²) // 深度學習 DP-SGD
差分隱私直覺理解
「任何人無法透過觀察演算法輸出，判斷某筆特定記錄是否在訓練集」。透過加入隨機噪聲實現。差分隱私 ≠ 加密（加密可解密；DP 的噪聲不可逆）。
▶ 考點：加噪聲隱藏個別記錄存在
DP-SGD（深度學習差分隱私）
梯度下降時：
1. 梯度裁剪：限制最大梯度範數 C
2. 加高斯噪聲：Noise~N(0,σ²C²)
3. 平均後更新參數
工具：Opacus（PyTorch）、TF Privacy
▶ DP-SGD = 梯度裁剪 + 高斯噪聲
聯邦學習（Federated Learning）
資料不集中到中央，模型在各裝置本地訓練，只上傳梯度。常與差分隱私結合：上傳梯度加噪聲，防止「梯度反轉攻擊」逆推原始資料。
▶ 考點：資料不動、梯度動
📌 考試必記：差分隱私核心 = 加入高斯或拉普拉斯隨機噪聲保護個別記錄。ε 越小保護越強但模型效用越低。與資料匿名化（k-匿名、概化）的區別：DP 提供數學可量化的隱私保護保證。
💻 程式碼考點（AI 系統部署相關 Python 工具）
模型量化（PyTorch 動態量化 FP32→INT8）
# PyTorch 動態量化：不需校正資料，最快速的量化方式
import torch
import torch.quantization

# 1. 載入已訓練的 FP32 模型並設為推論模式
model_fp32 = MyModel()
model_fp32.load_state_dict(torch.load('model.pth'))
model_fp32.eval() # 量化前必須設為 eval 模式

# 2. 動態量化（對 Linear 層量化為 INT8）
model_int8 = torch.quantization.quantize_dynamic(
  model_fp32,
  {torch.nn.Linear}, # 指定量化的層類型
  dtype=torch.qint8 # 目標精度 INT8
)

# FP32 大小 ÷ 4 ≈ INT8 大小（節省 75%）
print(f"INT8 節省約 {(1 - 1/4)*100:.0f}% 記憶體") # 輸出：75%
quantize_dynamic() vs quantize_static()
動態量化：不需校正資料集，直接量化 Weights，適合 NLP/LSTM。靜態量化：需要少量校正資料估計啟動值範圍，精度更好，適合 CNN/Vision。QAT（量化感知訓練）精度最佳但最耗時。
▶ 動態快、靜態準、QAT最佳
model.eval() 的重要性
量化前必須呼叫 .eval()，關閉 Dropout 和 BatchNorm 的訓練模式。否則量化時無法正確估計啟動值範圍（calibration 失敗），輸出不確定性高。
▶ 量化前必須 eval() 模式
dtype 精度選項
torch.qint8：有符號 INT8，-128~127（用於 Weights）
torch.quint8：無符號 INT8，0~255（用於 Activations）
torch.float16：FP16 半精度，省 50%，GPU 友好
▶ qint8 vs quint8 區別
FastAPI 部署 AI 推論服務（REST API）
# FastAPI 建立模型推論端點（現代 AI API 首選框架）
from fastapi import FastAPI
from pydantic import BaseModel
import numpy as np

app = FastAPI()

# 定義輸入資料結構（Pydantic 自動型別驗證）
class PredictRequest(BaseModel):
  features: list[float]

# POST 端點（推論傳資料用 POST，查詢用 GET）
@app.post("/predict")
async def predict(req: PredictRequest):
  X = np.array(req.features).reshape(1, -1)
  pred = model.predict(X)[0]
  return {"prediction": float(pred)}

# 啟動：uvicorn main:app --host 0.0.0.0 --port 8000
FastAPI vs Flask
FastAPI：自動生成 Swagger 文件、Pydantic 型別驗證、async/await（高並發）。Flask：簡單但無 async。iPAS 考試考 FastAPI 或 Flask 基本用法，理解 @app.post() 路由即可。
▶ FastAPI = 現代 AI API 首選
HTTP 方法語意
AI 推論 API 用 POST（需要傳遞特徵到 Request Body）。GET：查詢（無 Body）。PUT：更新。DELETE：刪除。考試常考：「推論端點應用哪個 HTTP 方法？」→ POST。
▶ 推論用 POST，查詢用 GET
BaseModel（Pydantic）自動驗證
繼承 BaseModel 定義 API 輸入結構，若傳入型別錯誤（如 features 傳字串），FastAPI 自動返回 422 Unprocessable Entity，不需手動寫驗證邏輯。
▶ 422 = 型別驗證失敗
Data Drift 偵測（scipy KS-test）
# Kolmogorov-Smirnov 雙樣本檢定偵測資料漂移
from scipy import stats
import numpy as np

# 訓練集特徵分佈（基準）
train_feat = np.random.normal(50, 10, 1000) # 均值=50
# 生產環境特徵（模擬漂移：均值從50→65）
prod_feat = np.random.normal(65, 15, 500) # 均值=65（漂移！）

# KS 雙樣本檢定
ks_stat, p_value = stats.ks_2samp(train_feat, prod_feat)
print(f"KS={ks_stat:.3f}, p={p_value:.4f}")

# p < 0.05 → 分佈顯著不同 → 觸發 CT 再訓練警告！
if p_value < 0.05:
  print("⚠️ Data Drift 偵測！建議觸發 CT 再訓練")
stats.ks_2samp()
比較兩個樣本是否來自同一分佈。ks_stat 越大（接近1）= 差異越大。p_value < 0.05 = 統計顯著差異 = 偵測到 Data Drift，應觸發再訓練。
▶ p < 0.05 = 漂移顯著
PSI（Population Stability Index）
PSI = Σ(實際比例 - 預期比例) × ln(實際/預期)
• PSI < 0.1：無漂移
• PSI 0.1~0.2：輕微（監控）
• PSI > 0.2：嚴重（再訓練）
金融業常用 PSI 監控信用模型。
▶ PSI > 0.2 = 嚴重漂移
np.random.normal()
生成常態分佈隨機數 (loc=均值, scale=標準差, size=樣本數)。模擬 Data Drift 時，改變 loc（均值偏移）或 scale（波動擴大），可測試偵測系統的靈敏度。
▶ 模擬 Data Drift 情境
