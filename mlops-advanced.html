① 資料漂移偵測（Data Drift Detection）
🔍 Data Drift vs Concept Drift ── 你錯了哪種？
Data Drift（特徵漂移）
輸入特徵 X 的分佈隨時間改變，但 X→Y 的關係不變。

例：顧客年齡分佈改變，但購買行為邏輯相同。
👉 Q22, Q32 考點
Concept Drift（概念漂移）
X→Y 的對應關係改變，相同特徵導致不同結果。

例：疫情後「出行頻率」對「消費力」的預測關係改變。
👉 Q46 考點
Model Drift（模型衰退）
模型準確率隨時間下降，通常由前兩者導致。是結果，不是原因。
👉 監控目標
📐 KL 散度（KL Divergence）── Q22 答案 D 的核心概念

KL 散度量化兩個機率分佈的差異程度，是確認 Data Drift 是否發生的直接工具。

// KL 散度公式（非對稱）
D_KL(P ‖ Q) = Σ P(x) · log( P(x) / Q(x) )

// P = 訓練時的特徵分佈
// Q = 生產環境中的特徵分佈
// 結果越大 → 漂移越嚴重 → 需重新訓練
⚠️ 考試陷阱：定期重新訓練（A）是「解決方案」，提升複雜度（B）、增加測試資料（C）都無法「偵測」分佈變化。只有 KL 散度才是確認漂移的診斷工具。
📊 PSI（Population Stability Index）── Q32 答案 D 的核心概念

PSI 是 Data Drift 的標準監控指標，能提前預警模型效能下滑風險。

PSI = Σ ( P_prod - P_train ) × ln( P_prod / P_train )

// P_prod = 生產環境各分桶比例
// P_train = 訓練時各分桶比例
PSI 判斷標準
PSI < 0.1
穩定
0.1 ≤ PSI < 0.2
輕微漂移
PSI ≥ 0.2
顯著漂移
💡 PSI ≥ 0.2 應觸發模型再訓練！PSI 是最具預測效力的早期預警指標，比 CPU/記憶體使用率、API 延遲更能提前發現問題。
監控指標監控對象能否預警模型衰退考試評分
PSI（Population Stability Index）輸入特徵分佈✅ 最佳選擇⭐⭐⭐
KL 散度特徵分佈差異✅ 可用（診斷工具）⭐⭐⭐
模型置信度分佈預測輸出⚠️ 間接反映⭐⭐

③ 漸進式部署策略（Phased Rollout）
🚀 四種部署策略完整比較 ── Q23 考點

Q23 情境：大型醫院導入輔助診斷 AI，需要「降低衝擊 + 風險可控 + 回饋可收斂」。

✅ 答案 A：漸進式上線
  • 從單一部門（放射科）開始
  • 收集真實反饋後逐步擴展
  • 降低整體風險，可快速回滾
  • 最符合醫療高風險場景
藍綠部署（Blue-Green）
  • 維持兩套相同環境
  • 瞬間切換流量，零停機
  • 出問題可立即回滾
  • 適合需要審計的金融/法規場景
金絲雀部署（Canary）
  • 先將少量流量（5%）導至新版本
  • 逐步增加比例
  • 仍有部分用戶受新版本影響
  • 適合 B2C 產品驗證新功能
影子模式（Shadow Mode）
  • AI 在後台並行運作
  • 不影響現有流程
  • 記錄 AI vs 人類決策差異
  • 零風險收集真實數據
⚠️ Q23 陷阱：C 選項「僅在夜班啟用」不符合「回饋可收斂」的要求（夜班樣本偏差）；D 選項全院同步提示會造成認知偏差風險。
④ 超參數搜尋三大方法 ── Q36 核心
🔬 Grid Search vs Random Search vs Bayesian Optimization
方法搜尋策略優點缺點適用場景
Grid Search（網格搜尋）窮舉所有組合系統化、不遺漏計算成本隨維度爆炸超參數少（≤3個）、計算資源充足
Random Search（隨機搜尋）隨機抽樣組合探索效率優於 Grid無法保證找到最佳超參數多、快速探索
Bayesian Optimization（貝葉斯）基於歷次結果動態調整⭐ 最智慧、效率最高實現複雜訓練代價高、超參數多
💡 Q36 答案 B（Grid Search）：題目描述「系統化測試多種模型設定」，關鍵字「系統化」對應 Grid Search 的窮舉特性。Bayesian（D）雖效率更高，但題目強調「系統化」，故 Grid Search 是最直接的選項。
⑤ 超參數調校的資料洩漏陷阱 ── Q14 & Q41
🚨 K-Fold + 超參數調校 → Data Leakage 問題（Q41）
❌ 問題根源：在 K-Fold 上直接調超參數 → 超參數針對整體資料分佈優化 → 過度樂觀偏差（Over-optimistic Bias）= Data Leakage！
✅ 正確解法：使用巢狀交叉驗證（Nested CV）
外層 CV：評估整體模型性能
內層 CV：選擇最佳超參數（隔離選擇過程）

巢狀交叉驗證（Nested CV）示意

外層 Fold 1（測試集隔離）
  └ 內層 K-Fold → 選出最佳超參數
外層 Fold 2（測試集隔離）
  └ 內層 K-Fold → 選出最佳超參數
  ... 重複
最終報告外層的平均性能 → 無偏估計！

🎯 交叉驗證最佳化泛化能力（Q14）
做法是否有效原因
A. 交叉驗證（Cross-Validation）✅ 最佳答案多組 train/val 切分評估，選出在驗證資料上最穩定的超參數
B. Early Stopping⚠️ 部分正確監控的是驗證誤差（不是訓練誤差）；屬訓練策略，非超參數搜尋方法
C. 特徵標準化❌ 無關資料前處理，改善數值穩定性，不直接解決超參數過擬合
D. 提高模型複雜度❌ 錯誤複雜度提升 → 更容易過擬合，方向相反
⑥ GPU 記憶體不足解決策略 ── Q37
🖥️ 大型模型訓練 GPU OOM 處理

情境：語音合成大型模型，模型架構固定、不能換硬體，需在維持效能前提下降低單 GPU 記憶體壓力。

✅ 答案 B：最有效
  • 較小的 Batch Size
  • + 資料分片（Data Sharding）
  • + 梯度累積（Gradient Accumulation）
  • 模擬大批次訓練效果
模型並行（Model Parallelism）
  • 將模型不同層分配到不同 GPU
  • 適合超大模型（單卡無法容納）
  • 題目說模型架構固定 → 較複雜
混合精度訓練（FP16）
  • 記憶體減少約 50%
  • 是常用優化技術
  • 但超大模型仍需搭配分散式策略
❌ 減少訓練資料（A）
  • 降低訓練效果，不是解決方案
  • ❌ 增加學習率（C）→ 加速不等於省記憶體
  • ❌ 用測試集訓練（D）→ 違背ML基本原則
// 梯度累積原理（模擬大 Batch Size）
// 實際 batch=8，累積 4 步 → 等效 batch=32
for step in range(accumulation_steps):
  loss = model(small_batch) / accumulation_steps
  loss.backward() // 累積梯度
optimizer.step() // 每 N 步才更新一次
optimizer.zero_grad()
